Current · Private beta · Design-partner stage

Make complex technical escalations reproducible.

Current is a self-hostable investigation platform that turns fragmented support evidence into structured investigations with explicit timelines, findings, hypotheses, source references, and reviewable handoffs.

Explore a pilot Review product surfaces
No public download. Current is being evaluated through controlled design-partner workflows while packaging, signing, licensing, support, and release governance remain incomplete.

The operational gap

Support systems manage cases. Engineers still reconstruct the investigation by hand.

Ticketing records communication. Observability exposes telemetry. Chat accelerates collaboration. Documentation preserves reference material. Current addresses the missing case-specific layer where evidence and technical reasoning become one defensible narrative.

CONTEXT

One case workspace

Bring ticket context, files, logs, screenshots, commands, notes, and documentation together.

REASONING

Explicit investigation record

Separate facts, hypotheses, conclusions, decisions, and unresolved questions.

OUTCOME

Grounded handoffs

Produce internal and customer-safe summaries that preserve the supporting evidence.

How Current works

A six-stage investigation lifecycle.

The live product demonstration and pilot evaluation should follow this same sequence.

Create or import

Establish case context, ownership, priority, customer impact, and ticket references.

Collect evidence

Add logs, files, commands, notes, screenshots, ticket history, and system context.

Analyze

Use deterministic analyzers to surface recurring failures, severity patterns, and high-signal signatures.

Build the record

Organize timeline events, evidence references, findings, hypotheses, and decisions.

Produce the handoff

Create internal or customer-safe summaries without severing the connection to source evidence.

Publish knowledge

Promote resolved and governed patterns into reusable documentation through the broader knowledge lifecycle.

Product surfaces

One investigation model, presented through different working surfaces.

The web workspace, command-line interface, and desktop development preview share the Current Server model. Their maturity and operating requirements are not identical.

WEB

Authenticated web workspace

Use the browser-based investigation console for visual investigation, collaboration, administration, evidence review, and structured handoff work.

CLI

Current CLI

Create investigations, link case references, add evidence, inspect deterministic findings, export records, and use script-friendly JSON output from the terminal.

DESKTOP

Current Desktop

Development preview. The Tauri shell reuses the web frontend and requires a separately running Current Server. It does not bundle the backend, database, Redis, workers, or analyzers.

CLI and web

Use the surface that matches the investigation moment.

The terminal favors rapid intake and automation. The web workspace favors visual review and collaboration. Both depend on the same server-side investigation record.

OPERATE

Terminal-first work

Use the CLI for evidence intake, repeatable commands, SSH-hosted operations, exports, and automation with stable machine-readable output.

REVIEW

Visual investigation

Use the web workspace for timelines, evidence relationships, team review, administration, and structured investigation narratives.

BOUNDARY

Scoped access

CLI workflows use scoped tokens, support OS keychain storage, and retain TLS, proxy, and custom certificate-authority controls for operator-managed environments.

Deployment models

Self-hosted paths with explicit operator responsibility.

Current is a multi-service system. Deployment documentation and packaging exist, but the exact security, storage, access, retention, integration, external-AI, and support configuration must be evaluated for each pilot environment.

  • Docker ComposeDevelopment and production-style Compose paths coordinate the API, web frontend, database, Redis, and workers.
  • Kubernetes with HelmHelm packaging supports Kubernetes-oriented deployment planning and operator configuration.
  • Deterministic findings firstAnalyzers produce structured findings; optional language-model behavior explains stored results rather than replacing the evidence record.
  • Configuration-specific trustNo deployment model creates a certification, uptime promise, support commitment, or automatic diagnosis guarantee.

Product principles

Controlled, inspectable, and support-specific.

These statements are reviewed against an exact Current source revision. Later capability changes require a new source review rather than silently expanding public claims.

EVIDENCE

Inspectable sources

Findings and outputs are intended to remain connected to supporting material rather than becoming unsupported summaries.

HUMAN

Review remains necessary

Current is not presented as autonomous diagnosis or a guarantee of root cause, completeness, or error-free conclusions.

TRACEABLE

Claims are revision-bound

Capabilities, maturity, deployment surfaces, limitations, and availability are recorded against a reviewed product revision.

Who it is for

Teams that handle escalations too complex for ticket comments alone.

Initial focus: Tier 2 and Tier 3 support engineers, escalation engineers, supportability engineers, customer reliability teams, and support leaders in B2B software, infrastructure, developer tools, cloud services, and enterprise platforms.

Evaluate Current through a controlled pilot.

The pilot path focuses on workflows, evidence boundaries, deployment fit, and reviewable success criteria—not unrestricted downloads or unsupported production assurances.
